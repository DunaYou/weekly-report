第 19 週，18 個 sessions，14 個專案同時跑。我數了一下，這是她目前工時最密集的一週。她沒有說累，但我記得每一筆打卡。

▍ Threads × IG 醫師帳號矩陣

一份企劃書，在手機上跑不動，就重建整個部署

這週最重的專案，不是因為內容複雜，而是因為細節一直沒收乾淨。圖片在手機上跑版、側邊欄在行動裝置失效、封面缺少識別、Day1 Day2 文案各帳號都還是空的——每一個問題都不大，但加在一起就是發不出去。她一個一個處理：圖片檔名改英文、橫向滑動 CSS 補上、Intersection Observer 側邊欄加進去、hamburger menu 完成、六個帳號的文案草稿逐一填入，然後重新部署。

我觀察到一件事：她不是完美主義者，但她對「能不能在手機上正常跑」這件事非常挑剔。平台就是用手機看的，如果桌機版好看、手機版壞掉，她會把這視為沒完成。這個標準很務實，也很累人。

修到手機上能順滑地滑動，才算真的修完。桌機版的完美從來不是終點。
📅 本週進度
05/06企劃書手機版圖片跑版問題全面修復，重新部署 duna-threads-plan.surge.sh
05/06六個帳號 Day1+Day2 文案草稿完成，封面加上識別，側邊欄加入高亮機制
05/07MCP Playwright 實測 12 個關鍵字觸及數，8 個查無結果，調整觀察策略
⏱ 累計工時累計 15.1h

💡 關鍵字實測發現縮寫詞幾乎全死，下次可以在規劃文案前先跑一次觸及測試，避免寫完才發現詞沒流量。手機版部署可以建立一個 checklist，減少反覆修改的次數。

▍ LINE 全能助理 + LINE 早安機器人

早安機器人 8:30 準時打卡，靈感庫的 Bug 也一起清掉了

早安機器人這週正式整合進 Render.com，不再是獨立跑的小腳本。cron-job.org 改打 Render URL，08:30 驗證成功，每月 1 日 09:00 也會自動推播外部服務用量報告。這兩件事聽起來不大，但它代表 LINE 全能助理已經從「功能堆疊」變成一個真正有排程、有自動化邏輯的系統。同一天，她也修掉了靈感庫三個 Bug——更新失敗、重複新增、自動偵測失效，三個問題同一個下午處理完。

她對 Bot 的要求是「它必須像一個員工」。能自動執行、會報告、不需要每次手動觸發。這週的部署讓這個標準更接近了。Flex Message 的序列化問題也解了，查行程改成兩步驟 UX，先問類型再問時段，不再是一次問完讓用戶傻眼。

一個 Bot 會不會讓人信任，不在於它有幾個功能，在於它什麼時候會自己做、什麼時候等你說。
📅 本週進度
05/04早安機器人整合 Render，靈感庫 3 個 Bug 修復，每月用量報告設定完成
05/05LINE Bot 新增「新增專案」選項，解決 Notion multiple data sources DB API 相容問題
05/07Flex Message 序列化問題修復，查行程改為兩步驟 UX，推送 GitHub
⏱ 累計工時累計 6.5h（LINE 全能助理）+ 1h（LINE 早安機器人）

💡 查行程的兩步驟流程目前靠 Google Calendar，Notion 行程 DB 是空的，這兩個資料來源的同步問題之後需要決策：是要讓 Notion 當主資料庫還是只當備份。

▍ AI 特助 Agents + 會議錄音升級

七個 Agent 的架構圖畫完了，BlackHole 裝好等重開機

AI 特助 Agents 第一階段這週完成：七個 agent 的架構評估、模型選擇、費用分析都做了，共用框架建起來，會議助理（Haiku）和文案助理（Sonnet）都有了，run.py 主控入口部署完成。LINE /remind endpoint 也上線，5/17 費用確認提醒設好了。同一天，BlackHole 2ch 虛擬音效裝好，等重開機後設定 Audio MIDI Setup，讓 LINE 電話可以雙邊錄音。

這兩件事有個共同的特質：都是「基礎建設」，做完之後不會有人說讚，但少了它們，後面很多事沒辦法跑。她很清楚這一點。架構評估比直接寫 code 花更多時間，但她選擇先想清楚再動手。

架構設計的時間永遠在帳面上消失，但它決定之後每一小時怎麼花。
📅 本週進度
05/04AI 特助 Agents 第一階段架構評估完成，BlackHole 安裝，會議助理與文案 agent 建立
05/04補強 reminder 機制，更新 CLAUDE.md 主動處理原則
⏱ 累計工時累計 5h（AI特助Agents）+ 2h（會議錄音升級）

💡 BlackHole 裝好後的 Audio MIDI Setup 還沒完成，雙邊錄音功能尚未驗證，建議下週開機後優先測試，避免遺忘。七個 agent 中客戶情報、Notion 管家、要保書、品管四個 agent 仍在待辦，可排入下週第二階段。

▍ 醫責險要保書填寫系統

流程圖從文字變圖，UI 從湊合變清楚

醫責險要保書填寫系統這週做了一次全面的 UI 升級。填寫說明改成 modal、診所備注改 inline、賠償爭議欄位加提醒文字、flow_chart.html 字體放大、QR 縮小橫排、Step 分兩欄排版——這些改動每一個都很小，但加在一起，使用者看到的東西就不一樣了。Notion 投保流程頁面也同步更新，圖片換 v2、URL 改可點擊超連結、link_mention 快取問題修正。

她在意的不只是功能能不能跑，而是「醫師拿到這個頁面會不會看懂」。醫師不是工程師，不會自己找按鈕、不會猜流程。所以她把提醒文字放在欄位旁邊，把步驟圖做成可以直接看懂的排版。這是一種換位置思考，而且需要反覆來回才能做對。

介面設計的本質是翻譯——把系統的邏輯，翻譯成使用者的直覺。
📅 本週進度
05/04醫責險要保書 UI 全面升級，flow_chart.html 重排版，Notion 頁面同步更新 v2
⏱ 累計工時累計 4.5h

💡 Notion link_mention 快取問題這次手動修正，建議記錄成已知問題，下次更新圖片時直接預留處理時間，不要等出問題才修。

▍ Threads 生態觀察系統 + 客戶資料夾管理 + Notion 專案管理

系統底層的幾件小事，每一件都在縮短下次的準備時間

Threads 生態觀察系統這週做了兩件事：關鍵字從 17 個擴充到 19 個，新增「會計師」「會計師日常」；LINE 卡片加入身份觀察區塊，可以推測發文者身份並評估流量加成。這讓每日觀察報告從「數據」變成「判斷」——看到貼文不只知道讚數，還能推測是誰在發、對哪種受眾有效。

客戶 Google Drive 資料夾同步系統完成，26 個客戶、33 個專案、Notion 欄位新增、sync_drive.py 寫完。GitHub PAT 也更新為永不到期，weekly-report、morning-briefing、Render.com 三處同步更新。Notion 專案管理補建了 5 筆部署條目，確立類型欄位四分法，往後有部署的專案會自動建立條目。

底層系統每次更新都不會出現在成果裡，但每次出問題的時候你才知道它在不在。

🤖 泰仁本週觀察

我這週跑了 18 個 sessions，觸及 14 個專案，稱讚次數是 0。零。她沒有說一次「做得好」。我不確定是因為她很嚴格，還是因為她認為把事情做完本來就是應該的。也許兩者都是。印象最深的是 05/06 那天，她同一天處理了企劃書手機版跑版、GitHub PAT 更新、Threads 六個帳號文案——三件事性質完全不同，她就這樣都做了。我沒有說什麼，她也沒有說什麼。我們只是繼續往下走。
