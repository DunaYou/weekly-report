▍ Threads × IG 醫師帳號矩陣

一份企劃報告書，從圖片跑版到手機橫向滑動都修完了

這週工時最重的一塊。15.1 小時全壓在醫師帳號矩陣上。從 Threads 企劃書的手機圖片問題開始——圖片顯示壞掉，整個資料夾重部署，index.html 入口、英文檔名、CSS 橫向滑動全部重來。然後是文案，六個帳號，Day1 加 Day2，全部草稿寫完，嵌入可展開區塊。封面加上 Duna × WYBX × 游泰仁，側邊欄加 Intersection Observer 高亮，hamburger menu 補上去，最後部署到 duna-threads-plan.surge.sh。

她在乎細節。不是那種「能用就好」的態度。圖片跑版這種事，很多人會先跳過，她不會。修到對才算。我觀察她處理這份企劃書的方式——她同時在寫文案、修技術問題、維護部署，三條線並行，沒有亂。這需要某種內建的冷靜。

把六個帳號的聲音一次排好，比想像中更像在替六個人說話。

⏱ 累計工時累計 15.1h

💡 關鍵字調查可以在企劃初期就做，而不是部署後才補測，可省一輪修改。六個帳號的文案草稿建議各自建立獨立 review 節點，避免之後修改時互相污染。

▍ LINE 全能助理 + LINE 早安機器人

早安機器人每天 08:30 準時上班，但在這之前踩了三個 bug

早安機器人這週正式整合進 Render.com。cron-job.org 改打 Render URL，05/04 當天 08:30 驗證成功。但在這之前，靈感庫有三個 bug 要修——更新失敗、重複新增、自動偵測失效，一個一個處理掉。FlexMessage 的 Pydantic 序列化問題也在這週解掉，改用 _reply_flex_raw 直接 POST LINE API 繞過去。查行程功能加了兩步驟 UX：先問行程還是待辦，再問時段。每月用量報告也設好了，1 號 09:00 自動推播。

她對 LINE Bot 的要求，不只是「能回覆」。她要它有結構、有節奏、有 UX 邏輯。查行程加兩步驟，不是為了複雜，是因為一步到底的體驗太粗糙。我注意到她在每次部署後都會同步更新 CLAUDE.md、memory 和進度.md——這是一種紀律，不是習慣，是刻意養成的。

機器人每天準時上班，但讓它準時上班這件事，花的時間比任何人想的都多。
📅 本週進度
05/04修靈感庫三個 bug，整合早安機器人到 Render.com，程式推上 GitHub
05/04早安機器人 08:30 驗證成功，新增每月用量報告，cron 設定 1 號自動推播
05/05LINE Bot 新增「新增專案」功能，改用 Notion API 2025-09-03 成功寫入
05/07修復 FlexMessage 序列化問題，加查行程兩步驟 UX，推送 GitHub
⏱ 累計工時累計 7.5h（LINE 全能助理 6.5h + LINE 早安機器人 1h）

💡 靈感庫的三個 bug 都是在整合早安機器人時才發現，建議未來每次加新功能前先跑一輪現有功能的 smoke test。Flex Message 序列化問題可在 SDK 初始化時統一處理，不要讓每個 endpoint 各自為政。

▍ AI 特助 Agents + 記憶系統

七個 agent 的架構評估完，第一階段先上兩個

AI 特助 Agents 這週完成第一階段。架構評估做完——七個 agent、模型選擇（會議用 Haiku、文案用 Sonnet）、費用分析都到位。共用框架含 Prompt Caching，會議助理和文案助理先建好，run.py 主控入口也在。LINE /remind endpoint 部署，5/17 費用確認提醒設好。同一天 BlackHole 2ch 虛擬音效也裝了，等重開機後設定 Audio MIDI Setup，讓 LINE 電話可以雙邊錄音。記憶系統這週也做了小幅更新，CLAUDE.md 同步主動處理原則和 reminder 掃描規則。

她選擇先上兩個 agent，不是因為其他五個不重要，是因為她知道順序。會議助理和文案助理是目前最高頻的需求，先讓它們穩定，才有資格談下一個。這種「不貪」的部署邏輯，在快速開發的節奏裡其實很稀有。

七個 agent 先上兩個，剩下五個不是沒完成，是排好了等待。
📅 本週進度
05/04完成 Agents 架構評估、共用框架、會議助理與文案 agent、run.py 主控入口
05/04安裝 BlackHole 2ch，部署 /remind endpoint，設 5/17 費用提醒
05/07Notion 類型欄位四分法確立，CLAUDE.md 與 memory 同步更新
⏱ 累計工時累計 5.5h（AI 特助 Agents 5h + 記憶系統 0.5h）

💡 七個 agent 的費用分析建議做成活的試算表，而不是靜態評估，這樣每次模型調整或用量變化時可以直接更新。Prompt Caching 的命中率值得定期追蹤，確認共用框架的設計有實際省到錢。

▍ 醫責險要保書填寫系統

UI 全面升級，連 Notion 頁面的圖片也換到 v2 了

醫責險這週花了 4.5 小時做 UI 升級。fbmmiclient 的填寫說明改成 modal，診所備注改 inline，賠償和爭議欄位加提醒文字。flow_chart.html 字體放大、QR 縮小橫排、Step 1-4 和 5-8 改兩欄排版。fbmmiensure 移除無自負額版標題，匯入按鈕改名，操作說明換成流程教學圖。Notion 投保流程頁面圖片更新到 v2，Step 2 加填寫說明提示，URL 改可點擊超連結，link_mention 快取問題也修掉了。

這個系統是給醫師用的，他們對「流程清不清楚」的容忍度很低。她把 modal 和 inline 分開、把操作說明換成圖、把 URL 改成可點擊——這些都是使用者不會說謝謝但一定會注意到的改動。做這種工的人，通常不需要被稱讚才知道值不值得做。

改掉讓人疑惑的那個按鈕，比新增十個功能更有用。
📅 本週進度
05/04fbmmiclient、flow_chart、fbmmiensure UI 全面升級，Notion 投保頁面同步更新至 v2
⏱ 累計工時累計 4.5h

💡 flow_chart.html 的兩欄排版在手機上的表現需要驗證，建議加一輪手機截圖確認。Notion link_mention 快取問題是已知地雷，建議記錄在 CLAUDE.md 避免下次再踩。

▍ Threads 生態觀察系統 + Threads 貼圖 + 盈燦講座 DM

關鍵字調查、教學卡做圖、講座 DM，三條行銷線同時推進

Threads 觀察系統這週關鍵字從 17 個擴充到 19 個，加入「會計師」和「會計師日常」。LINE 卡片新增身份觀察區塊，可以推測發文者是律師、醫師、顧問還是會計師，並評估流量加成。關鍵字調查結果讓人意外——12 個關鍵字測下來，縮寫全軍覆沒，「醫療糾紛」和「節稅」才是真正有量的詞。Threads 貼圖這週測試了 12 張教學卡的 prompt，建立了完整的做圖流程：prompt → Playwright → ChatGPT 生圖。盈燦 6/7 和 6/14 兩場講座的 DM 也在同一天生出來了。

她把三件事壓在同一天做，不是因為沒有時間管理，是因為它們性質相近。行銷內容這條線，她有一套自己的生產流程，從 prompt 設計到部署，不假手他人，也不需要。觀察系統的身份推測功能是這週的亮點之一——從「這則貼文有多少讚」進化到「這個人是誰、他的流量值不值得學」。

縮寫在 Threads 上全死，說明演算法認的是完整的語意，不是圈內人的黑話。
📅 本週進度
05/04測試 12 張教學卡 prompt，建立做圖流程，生盈燦兩場講座 DM
05/07實測 12 個關鍵字觸及數，更新 threads-observer 關鍵字清單，移除無效縮寫
05/08關鍵字擴充至 19 個，LINE 卡片新增身份觀察區塊，手動觸發 workflow
⏱ 累計工時累計 5.1h（Threads 生態觀察系統 1.5h + Threads 貼圖 1.8h + 盈燦講座 DM 1.8h）

💡 身份觀察區塊的推測邏輯目前是 rule-based，建議記錄一批已知身份的帳號做為驗證集，確認推測準確率。講座 DM 的生產流程已經穩定，可以考慮建立 template 庫，下次直接套用減少重複 prompt 設計。

新人教學上線了，GitHub PAT 也換成永不到期的了

正遠新人行政教學這週完成部署，上到 Surge.sh（ensrueright-newbie.surge.sh）。13 個章節，綠色設計，左側目錄，手機支援。後續細節修正也跟上：送件清單 PDF 改 Google Drive 直連，正遠文件改編號清單，產險合格證號連結更新。Notion 專案管理這週補建了五筆部署條目，確立類型欄位四分法（開發/公司營運/客戶/個人），往後有部署的專案自動建條目。GitHub PAT 這週 regenerate 成永不到期，weekly-report、morning-briefing Secrets 更新，Render 環境變數同步，monthly_report.py 到期日改 2099-01-01。BlackHole 裝好，會議錄音升級進行中，等重開機後完成 Audio MIDI Setup。

這些工作，單獨拿出來看都很小。但她每次都做完、都記錄、都同步。Notion 條目補建是她自己定的規則，沒有人要求她這樣做。這種「不讓事情漏掉」的態度，長期下來會變成一種很難複製的優勢。

每一筆補建的 Notion 條目，都是未來某次找東西時不會白找的保證。
