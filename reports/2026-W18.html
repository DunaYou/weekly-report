▍ LINE 全能助理

一個能講話、能記事、能查資料的 LINE 機器人，從零到上線

從 app.py 寫起，串 Notion、串 Google Calendar、串 GitHub Models。部署到 Render，設好 keep-alive，調好 Webhook。最後加了 8 個重置關鍵字、Quick Reply 按鈕、靈感庫分類、自然語言偵測。不是先規劃再做，是邊做邊長出來的。

我注意到她對「讓工具真的能用」這件事很執著。一個 KeyError 讓待辦功能壞掉，她沒有跳過，直接深進去修。查詢排序的 bug 也是。不是非修不可的問題，但她修了。這種習慣讓工具最後都是真的能用，不是能跑。

大多數工具死在「勉強能跑」那一步。她不讓它死在那裡。

💡 靈感庫與待辦的自然語言偵測邊界還不夠清楚，容易誤觸發。可以建一個簡單的意圖分類測試集，防止改 A 壞 B。

▍ 週報系統

週報從一個 HTML 檔，變成一個有風格、有封面、會自動跑的發布系統

這週週報系統經歷了三個階段：建起來、升級 AI、改風格。先是 GitHub Pages + generate_report.py + GitHub Actions，手動測試 Run #1 成功，26 秒完成，第一份週報上線。然後升級 AI 從 Haiku 到 Sonnet 4.6，修了 JSONDecodeError，改了 prefill 問題。再來風格大改版：封面圖、墨綠配色、短句敘事、動態導覽、字樣過濾。最後因為 Anthropic 餘額不足，改用 GitHub Models gpt-4o-mini，又成功跑起來。

她在改這個系統的時候，其實是在替未來的自己寫規格。TOC 錨點、上下篇導覽、稱讚率統計、AI 觀察標籤——這些沒有人要求，都是她覺得「這樣比較完整」加進去的。這種自己訂標準然後自己做到的方式，是她工作裡一個很穩定的特質。

週報是給自己看的，但她做得像要給很多人看一樣。
📅 本週進度
04/29週報系統完整建置：GitHub Pages、生成腳本、Actions 自動執行、Run #1 成功
04/29封面圖生成、墨綠配色、工時格、記憶同步、GH_PAT 到期提醒
04/29TOC 錨點修正、公開網址確認可分享

05/01文學風格改版、動態導覽按鈕、字樣修復、主頁清理
05/01格式大改版：每日歷程、工時格、AI 觀察、側邊欄 TOC
05/01改用 GitHub Models 修復 API 餘額問題、LINE 推播確認正常
⏱ 累計工時累計 24.7h

💡 AI 模型換過兩次（Haiku → Sonnet → gpt-4o-mini），每次都花時間修格式解析問題。可以把 JSON 解析邏輯抽成一個獨立函式並加單元測試，換模型時不用從頭 debug。

▍ LINE 早安機器人 + 早安特助 + 備份系統

每天早上 8:30，機器人準時送來今天要知道的事

早安機器人功能齊全：天氣、行程、待辦、財經新聞，用 GitHub Actions cron 觸發，但遇到 schedule+workflow_dispatch 重複推播的問題，改用 cron-job.org 外部觸發才解決。5 月 1 日發現三個 GitHub Secrets 遺失，補回去，加空值檢查，才讓機器人真正穩定運作。早安特助這週加了週一備份提醒，backup-todo.yml 每週一在 Notion 自動建待辦。備份系統本身也完整建立：backup.sh 打包所有設定、restore.sh 新機一鍵還原。

這三個工具放在一起看，是一套「開始一天之前把事情交代好」的基礎設施。她花了不少時間在確保這些工具不會默默失效——Secrets 遺失、重複推播、空值崩潰，都是那種很安靜地壞掉的問題。她把它們找出來修好了。

工具最怕的不是做不到，是不知道什麼時候它已經不工作了。
📅 本週進度
04/29早安機器人建置完成，修復重複推播，改 cron-job.org 觸發
04/29換機備份 backup.sh/restore.sh 建立，安全設定封鎖危險指令
05/01補回三個遺失 Secrets，加空值檢查，機器人恢復穩定
05/01早安特助加週一備份提醒，backup-todo.yml 自動建 Notion 待辦
⏱ 累計工時累計 LINE 早安機器人 8h + LINE早安機器人 3.5h + LINE早安特助 1.5h + 備份系統 3.5h + 安全設定 2h

💡 Secrets 遺失沒有主動告警，是推播失敗才被發現。可以在 Actions 腳本開頭加 Secrets 存在性檢查，失敗時直接發 LINE 通知。

▍ 磐石客戶資料整理

1252 張圖片、OCR、模糊比對，最後整理出 874 筆有用的資料

從 Google Drive 拿到 1252 張圖片，用 OCR 抓聯絡資料，精確比對命中 174 筆，Python SequenceMatcher 模糊比對補到 283 筆，最後依信心度拆成主表 874 筆（185 筆有手機）與待確認表 94 筆。之後用 Apps Script 寫入試算表，整理欄位順序，格式化手機號碼。再用 Playwright 自動建立 41 個依經手人分頁的試算表分頁。

這類資料整理工作沒有什麼成就感的峰值，是一步一步讓混亂變成可以用的東西。她做得很安靜，也做得很完整——連欄位順序、手機格式、信心度分層都整理好了，不是只把資料塞進去就算。

資料整理的終點不是「整理完」，是「別人拿到就能用」。
📅 本週進度
04/30OCR 抓取、精確比對 174 筆、模糊比對補至 283 筆、依信心度拆表
04/30189 筆寫入試算表、欄位整理、手機號碼格式化
04/30Apps Script 自動建立 41 個經手人分頁
⏱ 累計工時累計 磐石客戶資料整理 4.5h + 磐石舊客資料整理 0.5h

💡 模糊比對命中率 29.2%，剩下 70% 還沒有手機資料。可以評估是否用地址或診所名稱做第二層比對，提高覆蓋率。

▍ 醫責險要保書

醫師責任險的要保書，現在可以在手機上填、預覽、直接列印

這週把富邦產物醫師業務責任保險要保書做成線上填寫系統。全 7 頁 PDF 座標校準完畢，用 iframe 隔離解決列印空白頁問題，手機版用 transform:scale() 三模式介面，部署到 Netlify。核保確認後調整了幾個邏輯：移除要保人被保險人同一人選項、加繳費方式選擇、持卡人資料自動帶入。第二個版本（有自負額）的四欄勾選座標用灰度掃描修正，確立了「灰度掃描強制工作流程」當作所有圖像定位任務的標準方法。

填寫系統這種東西，細節決定好不好用。勾選框偏右 25px、信用卡頁在選繳費單時要自動隱藏、Tab 順序要對——這些不是主功能，但都是她去處理了。勾選定位方法論的確立，是這週比較低調但很實用的一件事。

一個表單填起來順不順，往往取決於那些沒人注意到的 25px。
📅 本週進度
04/29無自負額版全 7 頁座標校準完成，部署 Netlify
04/30依核保確認調整邏輯：繳費方式、持卡人自動帶入、移除不適用選項
05/01有自負額版勾選座標灰度掃描修正，確立圖像定位方法論，Netlify 更新
05/02新增客戶填寫頁 QR Code、部署 GitHub Pages、整理多方案比較
⏱ 累計工時累計 醫責險要保書 6.5h + 醫責險要保書填寫系統 3.5h

💡 勾選座標靠手動灰度掃描校準，換一張新 PDF 就要重來。可以考慮寫一個半自動的座標偵測腳本，輸入 PDF 輸出候選座標清單，人工確認後存成 config。

▍ 會議知識庫 + 班表自動入曆

128 筆會議記錄變成 48 頁知識條目，班表改一次不用再手動輸入

會議知識庫從 128 筆 Notion 記錄裡篩出 43 筆有內容的，建立 48 頁知識條目，每頁格式統一：核心知識點加應用情境，建立來源會議 relation，7 大分類完整覆蓋。中途因為重複跑損壞前 8 頁，全部修復。班表自動入曆這週做了 parse_schedule.py，用 Apple Vision OCR 辨識排班表圖片，確認後自動寫入 Google Calendar，5 月更新時改良了座標推算方法，從線性擬合改成實際觀測日期為基準，測試完全正確。

這兩件事表面上不像，但底層邏輯一樣：把原本要人工處理的東西，變成機器可以處理的結構。知識庫是讓會議記錄不只是記錄，班表工具是讓每個月的更新不用重新手動輸入。

把資料變成結構，這件事的價值，往往要等到三個月後才看得到。
📅 本週進度
04/29會議知識庫 48 頁建立完成，修復前 8 頁損壞，建立來源 relation
04/29班表自動入曆 parse_schedule.py 建置，OCR 辨識，自動寫入 Google Calendar
04/30班表自動入曆 5 月更新，座標推算改良，刪舊增新事件，測試完全正確
⏱ 累計工時累計 會議知識庫 6h + 班表自動入曆 2.5h

💡 知識庫 48 頁格式目前靠人工確認，多次重跑容易損壞。可以在生成前先比對已存在的頁面標題，只新增缺少的，不覆蓋已有的。
