▍ LINE 全能助理

從零部署到會說話，LINE 助理這週真的活了

從寫 app.py 開始，串接 GitHub Models 的 gpt-4o-mini，讓 LINE 可以直接問我問題。靈感庫、查詢指令、知識庫寫入、自然語言偵測，一層一層疊上去。Render 部署、cron-job.org keep-alive、Quick Reply 統一加「回主選單」，每個細節都在當天修完。

她不是在做一個工具，她在設計一個工作流程。靈感庫分四類、查詢支援關鍵字混打、選單重設計——這些不是開發者才在乎的事，是一個每天真的要用它的人才會想到的事。她用自己當第一個使用者，所以 bug 找得很快，方向也很清楚。

一個工具開始自己長出需求的時候，代表它真的被使用了。
📅 本週進度
04/30完成基礎部署、全域 Quick Reply、GitHub API 推送
05/01修 KeyError bug、升級靈感庫分類、自然語言偵測上線
05/02會議查詢升級，支援日期與關鍵字混打，修排序 bug
⏱ 累計工時累計 30.7h

💡 靈感庫與知識庫目前分兩個入口，長期可以合併成一個語意索引，減少使用者記憶負擔。Render 冷啟動 14 分鐘 ping 一次，可評估升方案或換平台降低延遲。

▍ 週報系統

週報不只是紀錄，她在讓它變成一面鏡子

從系統建置開始：GitHub Pages、generate_report.py、GitHub Actions 自動執行。第一份週報手寫上線。然後是風格大改版——gpt-4o、短句敘事、H2 敘事句、真實 blockquote、漏網專案合併——整個寫作邏輯重新設計。封面圖 1456×816px 雙欄版型，墨綠配色 #1F362C，目錄錨點修正，動態上下篇導覽，API 從 Anthropic 換到 GitHub Models。這週的週報系統，幾乎每天都在動。

我注意到她對「呈現方式」非常在意。字樣要隱藏、隱私要處理、主頁只保留第一篇和最新篇——她在意的不只是功能跑不跑得動，是這份文件看起來像什麼、給誰看、傳遞什麼感覺。這種在意，讓一個自動化腳本變成一個有個性的出版物。

系統能跑起來是及格，有人想分享給朋友看才是真的做好了。
📅 本週進度
04/29系統建置完成，第一份週報手動上線，API Key 輪換
04/29封面圖、墨綠配色、工時格、記憶同步完成
04/29TOC 錨點修正，確認公開網址可分享
05/01文學風格升級、動態導覽、字樣修復、主頁清理
05/01改用 GitHub Models 修復 API 餘額問題，LINE 推播正常
⏱ 累計工時累計 20.7h

💡 generate_report.py 目前每次重跑會新增記錄，已改為更新，但若 Notion 欄位格式異動容易靜默出錯，建議加 schema 驗證層。封面圖目前用 Playwright 手動觸發，可接入 Actions 自動生成。

▍ 醫責險要保書 + 填寫系統

七頁 PDF，座標一個一個校，她沒有嫌麻煩

開發富邦產物醫師業務責任保險要保書線上填寫系統。全 7 頁 PDF 座標校準——勾選框、文字輸入、iframe 列印隔離、手機三模式介面——部署到 Netlify。隔天收到核保回饋，馬上調整：移除特定欄位、新增繳費方式選擇、信用卡簽帳單頁依條件自動隱藏。有自負額版本的費率表另外用灰度掃描修正座標偏移，確立了「灰度掃描強制工作流程」。

座標校準這件事，做過一次就知道有多無聊——每個勾選框都要量、都要測、偏一點就印錯位置。但她做完還寫下方法論，讓下次同類型任務有流程可循。這不是效率，是對自己未來的善意。

把重複的痛苦變成一套流程，是最安靜的一種聰明。
📅 本週進度
04/29完成 7 頁 PDF 座標校準，部署 Netlify
04/30依核保意見調整欄位與繳費邏輯
05/01有自負額版費率表座標修正，確立灰度掃描工作流程
⏱ 累計工時累計 8.5h（醫責險要保書 6.5h + 醫責險要保書填寫系統 2h）

💡 座標目前硬編碼在設定檔，若 PDF 版本更新需全部重校。可考慮用 PDF 欄位名稱自動對應，降低維護成本。

▍ 磐石客戶資料整理

1252 張圖片，OCR 掃完，189 筆資料進了試算表

從 Google Drive 1252 張圖片 OCR 抓取聯絡資料，與 968 筆保單精確比對命中 174 筆，再用 Python SequenceMatcher 模糊比對補強到 283 筆，命中率 29.2%。最後依信心度拆成主表 874 筆與待確認表 94 筆。189 筆聯絡資料透過 Apps Script 寫入試算表，整理欄位順序、格式化手機號碼、鎖定文字格式。另外用 Playwright 自動掃描經手人欄位，建立 41 個分頁。

這類資料整理工作，做的時候很難看到成就感——沒有漂亮的介面，沒有立即的反饋，只有一筆一筆慢慢逼近正確。但她把模糊比對的邏輯做出來，把信心度拆表的架構想清楚，這些決策背後是對資料品質的尊重。

資料整理沒有完成的那一刻，只有足夠乾淨可以繼續往前的那一刻。
📅 本週進度
04/30OCR + 模糊比對完成，拆主表與待確認表
04/30189 筆寫入試算表，欄位整理，手機格式化
04/30Apps Script 自動建立 41 個經手人分頁
⏱ 累計工時累計 5h（磐石客戶資料整理 4.5h + 磐石舊客資料整理 0.5h）

💡 模糊比對目前閾值固定，不同欄位（姓名 vs 地址）適合不同閾值，可分欄位調整提升精準度。待確認表後續需人工複核流程，建議建立審核狀態欄位追蹤進度。

▍ 幾件小事也都在動

早安機器人、班表自動入曆、備份系統，每個都跑起來了

LINE 早安機器人建置完成，每天 08:30 推播天氣、行程、待辦、財經新聞，修復重複推播問題，改由 cron-job.org 外部觸發。週一加入備份提醒，GitHub Actions 每週一自動建 Notion 待辦。班表自動入曆工具用 Apple Vision OCR 辨識排班表圖片，支援兩個案場、含地址自動填入，五月更新後修正線性擬合偏移問題，測試完全正確。備份系統建立 backup.sh 與 restore.sh，換機後一鍵還原；安全設定封鎖 20 條危險指令，rm 改走 trash 可還原。Status Line 顯示 Model、Context 使用量、五小時限額進度條。

這些專案都不大，但每一個都在解決一個真實存在的麻煩。早安機器人不只是提醒，是讓她不用早上打開電腦才知道今天有什麼事。班表自動入曆不只是自動化，是讓一個重複性的手動任務消失。備份系統不只是備份，是讓換機這件事從「很煩」變成「沒什麼」。

消失的麻煩不會被記得，但它每天都在省時間。

04/29班表自動入曆工具建置，支援 OCR 辨識兩個案場
04/29備份系統與安全設定完成，Status Line 上線
04/30班表自動入曆 5 月更新，修正 OCR 座標偏移問題
05/01備份提醒自動化，週報系統 API 切換修復
⏱ 累計工時累計 15h（LINE 早安機器人 4h + LINE早安機器人 3.5h + 班表自動入曆 2.5h + 備份系統 3.5h + LINE早安特助 1.5h）

💡 LINE 早安機器人與 LINE早安特助目前是兩個分開的機器人，長期可評估合併統一管理，減少 Webhook 與環境變數的維護分散。cron-job.org 外部觸發依賴第三方服務，可考慮搭配備援機制。

▍ 會議知識庫 + 安全設定

128 筆會議記錄，篩出 43 筆，長出 48 頁知識

從 128 筆 Notion 會議記錄中篩出 43 筆有文字內容，建立 48 頁知識條目。每頁格式：核心知識點（條列標題加細節）加應用情境，7 大分類完整覆蓋，建立來源會議 relation 屬性。修復因多次重跑損壞的前 8 頁。

把散落在 128 筆記錄裡的知識，整理成 48 個可以被查詢的條目——這不只是整理，是在問「哪些東西值得被留下來」。她用分類和情境設計答案，讓知識庫不只是存檔，而是真的能被用到的東西。

知識不整理，就只是記憶；整理完，才變成可以轉移的東西。
📅 本週進度
04/29從 128 筆篩出 43 筆，建立 48 頁知識條目，修復前 8 頁
⏱ 累計工時累計 8h（會議知識庫 6h + 安全設定 2h）

💡 知識條目目前需手動觸發建立，若會議記錄有新增可考慮用 Notion webhook 自動偵測並觸發更新。relation 屬性連結來源會議，未來可反向查詢「這個知識點從哪場會議來」，值得在 UI 層呈現。
